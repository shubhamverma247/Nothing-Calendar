# DotCal — FINAL PACKAGE: 14 Features + Codex Prompts + Marketing

> Supersedes previous feature lists. This is the locked roadmap after keep/remove review.
> Removed: Focus Blocks, Working-days Calculator. Deferred: Travel Buffer, Day-Summary notification, Pinch-zoom, ICS preview.
>
> GLOBAL CONSTRAINTS (apply to every Codex prompt below):
> - Offline-only. No INTERNET permission, no cloud, no OAuth, no analytics.
> - Room schema is LOCKED: 5 existing tables only, NO new tables, NO new columns, NO migrations.
> - Where per-entity flags/data are needed beyond the schema, use the established
>   "side-store" pattern (JSON flag-store in app storage keyed by event ID — same
>   pattern as Ghost Events). Build ONE shared side-store utility; do not create
>   per-feature copies.
> - Manual DI (no Hilt/Koin). Follow existing constructor-injection conventions.
> - Pro gating: check `KEY_IS_PRO` from the existing DataStore prefs; locked features
>   deep-link to `dotcal://paywall`.
> - Jetpack Compose, dot-matrix / Nothing-inspired visual language, existing theme tokens only.

## 1.3 — Smart Quick Add v2 (Free) — FULL PROMPT

```
You are working on DotCal, an offline Android calendar app (Kotlin, Jetpack Compose,
Room with a LOCKED 5-table schema, manual DI, no INTERNET permission).

TASK: Upgrade the existing structured Quick Add with a natural-language parsing
layer ("Smart Quick Add"). Tier: FREE.

ARCHITECTURE:
- Create an on-device parser: input string -> ParsedEventDraft(title, startDateTime,
  endDateTime?, isAllDay, recurrenceRule?). Pure Kotlin, no network, no ML runtime —
  deterministic grammar + regex.
- The parser PRE-FILLS the existing Quick Add / event editor flow. Do not create a
  parallel event-creation path; reuse the existing creation use case so defaults,
  conflict warnings (A2), and calendar-set logic all apply automatically.

PARSING REQUIREMENTS (English + Hinglish):
1. Relative dates: "today", "tomorrow", "kal" (tomorrow), "parso" (day after),
   "next monday", "agle somvar", "in 3 days".
2. Absolute dates: "14 march", "march 14", "14/3", "14-03-2026".
3. Times: "1pm", "13:00", "sham 5 baje" (5pm), "subah 9" (9am), "9 baje".
   Ranges: "2-4pm", "2pm to 4pm" -> start+end.
4. Recurrence: "every monday", "har somvar", "every mon wed fri", "daily", "roz",
   "weekly", "monthly" -> map to the app's existing rrule format.
5. Duration: "for 2 hours", "2 ghante" -> end = start + duration.
6. All-day detection: date without time -> all-day event.
7. Title = remaining text after date/time/recurrence tokens are consumed, trimmed.
   Preserve names/capitalization ("Lunch with Rahul").
8. Ambiguity: if time matched but date missing -> assume today (or tomorrow if the
   time already passed today). If nothing parseable -> whole string is the title,
   flow behaves exactly like current Quick Add (graceful degradation).

UI:
- Quick Add input becomes a single free-text field. As the user types, show a live
  parse preview chip row below the field: [Tomorrow] [1:00 PM] [Repeats: none],
  each chip in dot-matrix styling using existing theme tokens.
- Tapping a chip opens the existing picker for that field (manual override always
  available). Misparse must never cost the user more taps than the old flow.
- Confirm -> existing save flow.

CONSTRAINTS:
- Locale-safe date parsing; respect the app's existing week-start and 12/24h
  settings.
- Parser must be a pure function library with NO Android dependencies so it is
  unit-testable on JVM.

TESTS (JVM unit tests, minimum):
- 30+ parse cases covering all rules above, including Hinglish inputs
  ("kal sham 5 baje dentist", "har somvar gym 7am", "mummy ka birthday 14 march").
- Past-time rollover ("1pm" typed at 3pm -> tomorrow 1pm).
- Degradation case: "random text" -> title only, no date.
```

## 1.7 — QR Event Share (Free) — FULL PROMPT

```
You are working on DotCal (offline Android calendar, Kotlin, Compose, locked Room
schema, manual DI, no INTERNET permission).

TASK: Let users share an event as a QR code that another DotCal user scans to
import. Tier: FREE (both sides) — this is a deliberate viral loop.

ENCODE (share side):
- Reuse the existing A3 "Share Event" ICS/text serialization. Compress the ICS
  payload (Deflate) and Base64-encode; prefix with "DOTCAL1:" as a format/version
  marker.
- Generate the QR locally with the ZXing core library ("com.google.zxing:core") —
  pure Java, no network, no Play Services. Do NOT use ML Kit here.
- If the payload exceeds reliable QR capacity (~1.5KB after compression), strip
  description/media fields first and warn "Shared without description".
- UI: "Share as QR" action in the event detail sheet -> full-screen dialog with the
  QR on a white card (QR needs contrast; keep the card white in both themes),
  event title below, DotCal wordmark at the bottom. Include "Save as image" and
  system-share actions.

DECODE (scan side):
- "Scan event QR" entry in the add-event menu.
- Use CameraX + ML Kit barcode scanning BUNDLED variant
  ("com.google.mlkit:barcode-scanning") so the model ships in the APK and works
  fully offline. CAMERA permission with a plain-language rationale.
- On a "DOTCAL1:" payload: decode -> reuse the EXISTING ICS import path (do not
  write a second parser) -> show the standard import preview -> save.
- Non-DotCal QR: "Not a DotCal event code" gentle error, keep scanning.
- Foreign but valid ICS text in a QR: import it anyway (bonus interop).

CONSTRAINTS:
- No new tables/columns. No network. Camera frames must never be stored.

TESTS: round-trip encode->decode->import equality for timed, all-day, and
recurring events; oversize payload fallback; malformed payload rejection.
```

```
You are working on DotCal (offline Android calendar, Kotlin, Compose, locked Room
schema, manual DI, no INTERNET permission).

TASK: Generate a human-readable "when I'm free" text for a chosen date range that
the user pastes into WhatsApp/email. Positioning: the offline Calendly. Tier: PRO
(KEY_IS_PRO; free users see the entry point, get the paywall sheet).

SHARED ENGINE (important):
- Build a FreeSlotEngine that computes free gaps from events in a date range.

- Inputs: date range, working-hours window (default 09:00–21:00, user-adjustable),
  minimum slot length (default 30 min), calendar-set filter (respect the currently
  active calendar set), option to treat ghost events (C6) as busy or free.
- All-day events: user toggle "block the whole day / ignore".

OUTPUT FORMAT:
- Grouped, compact, human text — e.g.:
    "My availability (Mon 13 – Fri 17 Jul):
     Mon: 2:00–5:30 pm
     Tue: free all day
     Wed: before 11 am, after 4 pm
     Fri: 1:00–3:00 pm"
- Rules: merge adjacent slots; "free all day" when the whole window is open;
  "before X / after Y" when a single event splits the day; omit fully-busy days
  with a trailing "Other days are fully booked."
- Respect the app's 12/24h setting.

UI:
- Entry: "Share availability" in the calendar overflow menu + week-view long-press.
- Config sheet: date-range picker (presets: Next 3 days / This week / Next week),
  working-hours slider, min-slot selector, live preview of the generated text.
- Actions: Copy, and system Share.

TESTS (JVM): slot computation with overlapping events, all-day events, events
crossing midnight, empty calendar ("free all day" x N), full-day busy;
text formatting for each rule above.
```

## 1.14 — Vault Decoy PIN (Pro) — FULL PROMPT

```
You are working on DotCal (offline Android calendar, Kotlin, Compose, locked Room
schema, manual DI). DotCal has an existing PIN-protected Vault for private events.

TASK: Add a DECOY PIN — a second PIN that opens an empty (or separate) vault, for
plausible deniability. Tier: PRO.

BEHAVIOR:
- Vault settings (accessible only from inside the real vault) gain "Set decoy PIN"
  (KEY_IS_PRO gated).
- Decoy PIN must differ from the real PIN; validate on set.
- Entering the REAL PIN at the vault lock screen -> real vault (unchanged).
- Entering the DECOY PIN -> a fully functional but SEPARATE vault space that starts
  empty. Events created inside the decoy vault persist to the decoy space (side-store
  namespace "vault_decoy" for the membership flags, reusing the shared utility from
  prompt 1.2 — no schema change).
- CRITICAL: the decoy vault must be visually IDENTICAL to the real one. No label,
  toast, log line, or settings difference visible from inside the decoy. Decoy
  settings show "Set decoy PIN" too (no-op that appears to work) so inspection
  reveals nothing.
- Wrong PIN (neither real nor decoy) -> normal error path, unchanged.
- PIN storage: hash both PINs with the same KDF the vault already uses (never
  plaintext). If the vault currently stores a weaker hash, upgrade both to a salted
  KDF now.
- Removing the decoy PIN (from the real vault only) deletes decoy-space membership
  flags after a confirm dialog.

TESTS: PIN routing (real/decoy/wrong); decoy isolation (decoy session can never
query real-vault events — verify at the repository layer, not just UI); flags
round-trip; KDF verification.
```
